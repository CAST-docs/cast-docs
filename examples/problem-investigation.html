Updated 2026-05-13

Background背景

  • Scope范围: Checkout requests showed intermittent latency spikes.Checkout 请求出现间歇性延迟尖峰。
  • Window时间窗口: Observed during the latest traffic peak.在最近一次流量高峰期间观察到。

This document records the investigation path for the tail-latency regression and the follow-up actions.本文记录 尾延迟 回归的排查路径和后续动作。

Problem Symptoms问题症状

Symptom症状

P95 latency jumped from 310ms to 1840ms during peak, while success rate held near 99.2%.P95 延迟从 310ms 上升到 1840ms，发生在高峰期间，而成功率保持在约 99.2%.

Investigation Path排查路径

Core Logs And Evidence核心日志与证据

Representative log代表性日志
checkout latency=1840ms dependency=inventory status=200
  • services/checkout/inventory_client.go:88 - Dependency call without timeout override依赖调用缺少超时覆盖

Root Cause根因

Root cause根因

Checkout inherited the 30s global dependency timeout instead of the intended 3s; the per-call override in inventory_client.go:88 was never wired up.Checkout 继承了 30s 的全局依赖超时，而不是预期的 3s；位于 inventory_client.go:88 的单次调用覆盖从未接入。

Mitigation And Fix缓解与修复

Set checkout-specific timeout设置 Checkout 专用超时

Wrap the reservation call in a context.WithTimeout of 3s, replacing the inherited DefaultTimeout. Track rollout in the checkout timeout runbook.用 context.WithTimeout 包裹预占调用，超时时间为 3s，替换继承而来的 DefaultTimeout。在 Checkout 超时 runbook.

Inspect checkout inventory calls and propose a timeout change with tests.检查 checkout 的库存调用，并提出带测试的超时修改方案。

Validation验证

  1. P95 checkout latency returns below 400ms.P95 checkout 延迟恢复到 400ms.
  2. Timeout behaviour is covered by unit tests on reserveInventory.针对 reserveInventory.

Prevention Measures预防措施

  • Add a per-call timeout to the service template so new clients can't inherit the global default.在服务模板中加入单次调用超时，避免新客户端继承全局默认值。
  • Alert when dependency P95 exceeds 500ms, well before checkout-api regresses.当依赖 P95 超过 500ms 时告警，早于 checkout-api 发生回归。
