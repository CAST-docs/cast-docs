Checkout latency spike investigation

Updated 2026-05-13

Background

  • Scope: Checkout requests showed intermittent latency spikes.
  • Window: Observed during the latest traffic peak.

This document records the investigation path for the tail-latency regression and the follow-up actions.

Problem Symptoms

Symptom

P95 latency jumped from 310ms to 1840ms during peak, while success rate held near 99.2%.

Impact Scope

AreaImpactConfidence
checkout-apiP95 +1530mshigh
payment-callbackNo direct impactmedium

Investigation Path

Core Logs And Evidence

Representative log
checkout latency=1840ms dependency=inventory status=200
  • services/checkout/inventory_client.go:88 — Dependency call without timeout override

Root Cause

Root cause

Checkout inherited the 30s global dependency timeout instead of the intended 3s; the per-call override in inventory_client.go:88 was never wired up.

Mitigation And Fix

Set checkout-specific timeout

Wrap the reservation call in a context.WithTimeout of 3s, replacing the inherited DefaultTimeout. Track rollout in the checkout timeout runbook.

Inspect checkout inventory calls and propose a timeout change with tests.

Validation

  1. P95 checkout latency returns below 400ms.
  2. Timeout behaviour is covered by unit tests on reserveInventory.

Prevention Measures

  • Add a per-call timeout to the service template so new clients can't inherit the global default.
  • Alert when dependency P95 exceeds 500ms, well before checkout-api regresses.

Open Questions

  • Should inventory-svc expose a faster reservation endpoint for checkout-api?
  • Do the Prevention Measures cover the batch-reservation path too?
